> **참조**: [cc/docs/260518-002-DOCS-map api별 제공정보.md](../docs/260518-002-DOCS-map%20api별%20제공정보.md), [cc/api/v1/place.md](../api/v1/place.md), [cc/api/v1/plan.md](../api/v1/plan.md), [cc/api/v1/search.md](../api/v1/search.md)

## 1. 기존 장소 API 현황 요약

[cc/api/v1/place.md](../api/v1/place.md)와 인접한 [plan.md](../api/v1/plan.md), [search.md](../api/v1/search.md)에서 정의된 자체 API는 다음과 같다.

| 코드 | 메서드 | URL | 역할 |
|---|---|---|---|
| PLACE-4 | GET | `/api/place/search?keyword=` | 키워드 장소 검색 (자체 DB) |
| PLACE-16 | POST | `/api/place/recommend` | 설문 기반 추천 (출발지 좌표 `x,y` 전달) |
| PLACE-19 | POST | `/api/plan-place/{planId}` | 플랜에 장소 추가 |
| PLACE-N1 | GET | `/api/place/{placeId}` | 장소 상세 (`x, y, address, businessHours, …`) |
| PLACE-N2~N3 | POST/DELETE | `/api/place/like` · `/scrapped` | 좋아요·스크랩 |
| PLACE-N4~N5 | GET/POST | `/api/place/{placeId}/review` | 방문자 후기 |
| PLACE-N6 | GET | `/api/place/{placeId}/plan` | 이 장소가 포함된 코스 |
| SEARCH-1~3 | GET | `/api/search`, `/autocomplete`, `/trending` | 통합 검색·자동완성·인기 키워드 |
| PLAN(다수) | — | `/api/plan/*`, `/api/plan-place/*` | 플랜 생성/수정/장소 순서/스탬프 등 |

**관측된 데이터 모델 핵심 필드**

- 위치: `x`(경도), `y`(위도) — WGS84로 추정
- 주소: `roadAddressName`(도로명)
- 메타: `placeName, categoryName, placeImageUrl, businessHours, phoneNumber, tags, averageRating, reviewCount`
- 이미지 URL: `lh3.googleusercontent.com/place-photos/…` (Google Place Photos 사용 흔적)
- 코스 생성 입력: `startingPoint = { address, x, y }`, `transport`(도보/대중교통/자가용), `transportTime`
- 스탬프(방문 인증): `location: { x, y }` 클라이언트가 전달 → 서버에서 거리 검증(422 "해당 장소와 거리가 멀어요")

**즉, 자체 API가 이미 책임지는 것**

- 장소 ID 체계, 장소 메타(영업시간/카테고리/태그/이미지/평점)
- 장소 검색(키워드), 자동완성, 인기 키워드, 통합 검색
- 좋아요/스크랩/후기

**자체 API가 책임지지 않는 것 (= 외부 지도 API가 필요한 영역)**

1. **지도 렌더링** — 코스 미리보기, 장소 상세, 추천 결과를 지도 위에 마커로 표시
2. **사용자 입력 주소 → 좌표** — `startingPoint.address` 텍스트 입력을 받을 때
3. **현 위치 좌표 → 주소** — 출발지 자동 입력, 스탬프 위치 라벨링
4. **경로/소요시간** — `transport` 별 출발지→첫 장소→다음 장소 경로 표시 및 ETA
5. **위치 검증 거리 계산** — 스탬프 422 응답 기준점(보조)
6. (선택) **자체 DB 외 장소 보강 검색** — 사용자가 검색했는데 자체 DB에 없는 경우 외부 POI로 fallback

## 2. 외부 지도 API별 적합도 평가

[cc/docs/002](../docs/260518-002-DOCS-map%20api별%20제공정보.md) 기준으로 정리.

### 2-1. Kakao Local API

- **강점**: 국내 주소·POI 정확도 최상, 키워드/카테고리 검색·지오코딩 모두 한 SDK로 처리. 무료 쿼터가 사용 가능 범위에서 가장 너그러움.
- **약점**: 지도 *렌더링*은 별도 Kakao Maps JS SDK가 필요(Local API 자체는 데이터만). 경로(길찾기) 미제공 — 도보/대중교통/자가용 경로는 Kakao Local만으로 못 구함.
- **적용 후보**:
  - 1-1 주소→좌표 (Geocoding)
  - 1-3 좌표→주소 (Reverse Geocoding)
  - 1-2 좌표→행정구역 (장소 카드의 `location: "서울 성동구"` 형태 생성에 활용)
  - 1-5 키워드 장소 검색 (자체 DB 미스 시 fallback)

### 2-2. Naver Maps API

- **강점**: 국내 지도 렌더링 품질·라이선스 안정성 우수. Dynamic/Static Map 모두 제공. Reverse Geocoding 결과가 행정구역 트리(시·도/시·군·구/읍·면·동)까지 깔끔. Directions 5/15 자동차 경로 제공.
- **약점**: 보행자/대중교통 경로 미제공(자동차만). POI 검색 자체는 약함(주로 지도+지오코딩+경로 라인업).
- **적용 후보**:
  - 2-1 Dynamic Map — **메인 지도 렌더링** (코스 라인, 장소 마커, 사용자 인터랙션)
  - 2-2 Static Map — 코스 카드 썸네일·공유 OG 이미지 (트래픽 절약 큼)
  - 2-3 / 2-4 Geocoding / Reverse Geocoding (Kakao 대체 가능)
  - 2-5 / 2-6 Directions (자가용 transport 한정)

### 2-3. Google Places API

- **강점**: 사진/평점/리뷰/영업시간 필드가 압도적으로 풍부. AI 요약·접근성·결제·반려견 허용 등 메타 다양.
- **약점**: 국내 POI 커버리지·정확도가 부족. 한글 카테고리·도로명 누락 빈번. 호출 비용이 4종 중 가장 비싸고 Photos URI는 단발성 만료(캐싱 어려움).
- **현 사용 흔적**: PLACE-N1 `placeImageUrl`이 `lh3.googleusercontent.com/place-photos/...` → 백엔드가 이미 Google Place Photos로 일부 이미지를 수급 중인 것으로 추정.
- **적용 후보**:
  - 3-4 Place Photos — **백엔드 한정**, 자체 DB 이미지 수급 보조용
  - (프론트엔드 직접 호출은 비용·국내 정확도 모두 불리하므로 비권장)

### 2-4. TMAP API

- **강점**: **보행자/자동차 경로**를 모두 제공(4-15, 4-16). 다중 경유지·경유지 최적화(4-19, 4-20)는 코스 시간 계산에 직결. 장소 혼잡도(4-26), 직선거리(4-18)도 유용. 국내 도로/POI 데이터 우수.
- **약점**: 지도 렌더링은 가능하나 시각 디자인·SDK 성숙도는 Naver/Kakao 대비 부족. 일일 호출 제한·요금 정책이 가장 변동 잦음.
- **적용 후보**:
  - 4-15 자동차 경로안내 — `transport=자가용`
  - 4-16 보행자 경로안내 — `transport=도보`
  - 4-18 직선거리 — **스탬프 위치 검증** (PLAN 스탬프 422의 거리 임계값 계산, 백엔드용)
  - 4-19 / 4-20 다중 경유지/경유지 최적화 — `POST /api/plan-place/duration` (코스 시간계산·프리뷰)의 ETA 정밀화
  - 4-26 장소 혼잡도 — 추천 결과 카드의 보조 시그널(향후 확장)

> **대중교통 경로**는 4종 모두 직접 제공하지 않는다. `transport=대중교통`은 우선순위에서 후순위로 두거나, ODsay/Kakao Mobility 등 별도 API 도입을 추후 검토.

## 4. 자체 API 명세에 반영해야 할 변경 제안

기존 `cc/api/v1/place.md`·`plan.md`는 외부 API 통합을 명시하지 않은 상태다. 다음과 같이 보강·신설을 권장한다.

### 4-1. 신설(혹은 명시화) 필요 엔드포인트

| 신규 코드(안) | 메서드/URL | 목적 | 외부 API |
|---|---|---|---|
| PLACE-N7 | `GET /api/geo/geocode?address=` | 출발지 주소 → 좌표 | Kakao 1-1 |
| PLACE-N8 | `GET /api/geo/reverse?x=&y=` | 좌표 → 주소·행정구역 | Kakao 1-3 + 1-2 |
| PLACE-N9 | `GET /api/geo/keyword?query=&x=&y=` | 자체 DB 없는 외부 POI 검색 결과(가공) | Kakao 1-5 |
| PLAN-NX | `POST /api/plan-place/route` | transport별 경로 라인·세그먼트 ETA 반환 | TMAP 4-15/4-16/4-19 |

> 프론트엔드가 외부 API를 직접 호출하기보다 **백엔드(또는 Next.js API Route)로 일원화**하는 것이 키 노출 방지·요청 캐시·요금 통제 측면에서 안전.

### 4-2. 기존 응답 보강

- **PLACE-N1 장소 상세**: 응답에 `mapMeta`(예: `{ provider: 'naver', center: {x,y}, zoom: 16, markers: [...] }`) 또는 단순 `staticMapUrl` 추가 → 클라이언트가 마커 좌표 일관성 보장.
- **`POST /api/plan-place/duration`**: 응답 `places[].duration`을 **TMAP 경로 기반으로 산정**하도록 백엔드에서 외부 호출 후 결과 캐시. transport에 따라 다른 API 호출.
- **스탬프 422 응답**: 거리 임계값(예: `100m`)을 클라이언트에서도 표시할 수 있게 응답 본문에 `distanceMeters`, `thresholdMeters` 포함 권장.
- **검색/추천 응답의 `location`**: 좌표가 있는 항목은 백엔드에서 Kakao 1-2로 행정구역명을 미리 채워 응답 (지금처럼 `"서울 성동구"` 문자열).

## 5. 통합 방식 권장(아키텍처)

```
[Browser]
  └── Next.js API Route (src/app/api/*)   ← 키 보관·캐시·요금 게이트
        ├─ /api/geo/* ─────────► Kakao Local
        ├─ /api/plan-place/route ─► TMAP Directions
        └─ /api/place/search (fallback) ─► Kakao 키워드
[Browser, 지도 렌더링만]
  └── @vis.gl/react-google-maps 류 대신 → Naver Maps JS SDK 직접 로드
```

**핵심 원칙**

1. **데이터 API는 백엔드 경유**, **지도 렌더링 SDK는 클라이언트 직접 로드** (Naver Maps는 클라이언트 키 사용이 정상 사용 모델).
2. **이미지/지오코딩 결과는 캐시 가능한 값** → 백엔드가 자체 DB 컬럼으로 보관(`administrative_area_name`, `road_address`, `lng/lat`)하고 외부 호출은 신규 데이터 적재 시에만 발생하도록.
3. **transport 분기 책임은 백엔드** — 프론트는 `{transport, places: [...]}`만 던지고, 어떤 외부 API를 호출할지는 BFF가 결정.
4. **Google Places는 프론트에서 호출하지 않음** — 운영 적재 파이프라인 한정.

## 6. 우선순위 로드맵

| Phase | 범위 | API |
|---|---|---|
| **P0 (즉시)** | 출발지 입력 자동완성/현 위치 라벨링, 장소 상세 지도, 코스 상세 지도 | Kakao Geocoding 1-1/1-3, Naver Dynamic Map 2-1 |
| **P1** | 코스 카드 썸네일(공유 포함), 코스 시간 계산 정밀화(도보/자가용) | Naver Static Map 2-2, TMAP 4-15/4-16, 4-19 |
| **P2** | 자체 DB 미스 fallback 키워드 검색, 스탬프 거리 임계 응답 표준화 | Kakao 1-5, TMAP 4-18(또는 서버 자체 Haversine) |
| **P3 (백로그)** | 장소 혼잡도, 대중교통 경로(별도 벤더) | TMAP 4-26, ODsay 등 |

## 7. 한 줄 결론

- **지도 렌더링은 Naver**, **국내 주소·POI는 Kakao**, **경로/시간은 TMAP**, **Google은 백엔드 한정 이미지 수급용**으로 역할 분리하고,
- 프론트는 Next.js API Route를 통해 한 곳에서 호출하도록 BFF를 거치며,
- 외부 호출 결과는 가능한 만큼 자체 DB·캐시로 끌어와 `cc/api/v1/place.md` 응답 모델 안에 미리 박아 두는 것이 비용·정확도·UX 셋 다 가장 유리하다.
