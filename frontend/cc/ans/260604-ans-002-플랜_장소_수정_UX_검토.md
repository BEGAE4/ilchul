## 1. 플랜 장소 수정 — 일괄 vs 개별

### 결론: 개별 수정 + 편집모드 토글 방식 채택

**근거**
- 현재 모든 편집 UX(순서 변경, 추가, 삭제)가 개별 단위로 설계되어 있음 → 일관성 유지
- 모바일 환경에서 일괄 수정은 다중 선택 UI 필요 → 복잡도 증가
- 여행 플랜 특성상 특정 장소 하나만 바꾸는 케이스가 대부분
- 힐링 감성 앱 특성 → 단순하고 가벼운 UX가 적합

### 적절한 수정 자유도

| 기능 | 방식 |
|---|---|
| 시간대 / 메모 수정 | 장소 탭 → 개별 편집 |
| 장소 교체 | 장소 탭 → 다른 장소로 스왑 (개별) |
| 순서 변경 | 현행 유지 (위/아래 버튼 또는 드래그) |
| 다중 삭제 | 편집모드에서 일괄 허용 (자연스러운 예외) |

**일괄 수정이 의미 있는 시점**: "날짜 전체 이동" 같은 케이스 — 현 단계에서는 오버스펙

## 2. 완료된 코스(`phase === 'after'`) 편집 제한 현황

### 현재 구현 위치

`src/features/my-course/components/MyCourseDetailPage.tsx`

```ts
const canEdit = phase !== 'after';   // after이면 편집 불가
const canVerify = phase === 'during'; // during일 때만 인증 가능
```

### 기능별 허용 여부

| 기능 | after 상태 |
|---|---|
| 플랜 이름 수정 | 불가 (메뉴에서 숨김) |
| 플랜 순서 편집 | 불가 (메뉴 + 버튼 모두 숨김) |
| 플랜 복제 | **가능** (canEdit 조건 없이 항상 노출) |
| 플랜 삭제 | **가능** (항상 노출) |
| 위치 인증 | 불가 (canVerify = false) |

안내 문구: `'여행 기간이 종료되어 인증 및 수정이 불가합니다.'`

### 개선 여지

`플랜 이름 수정`은 완료 후에도 허용해도 무방한 기능이나 현재는 `canEdit`으로 일괄 차단됨.
필요 시 이름 수정만 `canEdit` 조건에서 분리 고려.
