## 0. 한 줄 결론

> 질문에 대한 답: **"기본 5, 쿼리에 따라 조절"은 맞지만, 그것만으로는 "더보기"를 지탱할 수 없다.**
>
> 현재 메인 4종 엔드포인트는 `limit`만 있는 케이스(MAIN-57·61)와 `limit + page`만 있는 케이스(MAIN-58·60)가 섞여 있고, **모두 응답에 페이지네이션 메타데이터(`hasNext`, `totalCount`)가 없다.** 더보기 UX를 제대로 지원하려면 4개 엔드포인트의 페이지네이션 규약을 통일하고, 응답에 메타를 추가해야 한다.

## 2. 핵심 문제점

### 2-1. MAIN-57·MAIN-61에는 `page`가 아예 없다

`/api/plan/popular`와 `/api/place/popular`는 `limit`만 받는다. 즉 **"limit을 키워 한 번에 더 가져오기" 외에는 추가 조회 수단이 없다.** 21번째 항목을 보고 싶다면 `limit=20` 뒤에 `limit=40`을 다시 보내는 식이라, 다음과 같은 문제가 생긴다.

- 매 호출마다 이미 본 데이터를 다시 받아와 트래픽이 누적된다.
- 실시간 랭킹 특성상 호출 사이에 순위가 바뀌면 **중복·누락**이 발생한다.
- 클라이언트에 캐싱·머지 로직이 없으면 스크롤 위치/스켈레톤 처리가 까다로워진다.

반면 MAIN-58·60은 `page`가 정의돼 있어 비교적 정상적인 페이지네이션이 가능하다.

### 2-2. 4개 엔드포인트 모두 응답에 메타가 없다

`hasNext`도 `totalCount`도 없으니 클라이언트는 다음 두 가지를 **추측**할 수밖에 없다.

- "끝에 도달했는지" — 응답 배열이 `limit`보다 짧으면 끝, 같으면 다음이 있을 수도 있고 없을 수도 있다 (휴리스틱).
- "전체가 몇 개인지" — 알 수 없다 → "총 N개의 플랜" 같은 헤더 카운트, "더 이상 결과가 없습니다" 종료 처리 모두 불가.

`PLACE-N4 장소 후기`가 이미 `hasNext` + `totalCount`를 노출하고 있으므로 같은 형태로 통일하면 된다.

### 2-3. 현재 홈 화면의 "더보기"는 `/search`로 보내는 임시 처리다

[src/app/page.tsx](src/app/page.tsx)의 4개 섹션 더보기 버튼은 모두 `router.push('/search')`로만 연결돼 있다. 즉 지금은 **API의 페이지네이션을 사용할 진입점 자체가 없는 상태**다. 명세의 "기본 3, 더보기 진입 시 사용"이라는 주석은 클라이언트가 아직 따라가지 못하고 있다.

따라서 "limit을 동적으로" 논의를 결론짓기 전에, 더보기 도착지(전용 리스트 페이지 vs 검색 결과로 흡수)를 먼저 정해야 한다.

## 3. 질문에 대한 직접 답변

> **"기본 5로 두고, 쿼리에 따라 조절하게 하는 게 나을까?"**

**그 자체는 맞다 — 그리고 사실 이미 그렇게 명세돼 있다 (MAIN-57은 `limit` 기본 5, 클라이언트가 보내면 그 값으로).**

다만 그 한 가지 변경만으로는 더보기를 지원하지 못한다. 두 가지 시나리오 중 하나를 선택해야 한다.

### A안. **전용 더보기 페이지 + `page` 파라미터** (권장)

- 메인: `?limit=5` 호출, 5건 캐러셀.
- 더보기 클릭 → `/popular/plan` 같은 신규 라우트로 이동.
- 더보기 페이지: `?limit=20&page=1`, 무한스크롤 또는 페이지네이터로 `page`를 증가시키며 호출.
- 명세 변경: **MAIN-57·MAIN-61에 `page` 파라미터 추가 + 4개 엔드포인트 응답에 `hasNext`·`totalCount` 추가**.

장점: 기존 MAIN-58·60과 컨벤션이 통일되고, "내 주변 + 위치 기반 + 페이지네이션"의 조합이 자연스럽다. 메인 화면 호출은 가벼운 5건으로 유지된다.

### B안. **메인 화면 인라인 더보기 (`limit`만 키우기)**

- "더보기" = 같은 섹션에서 `limit=10`, `limit=15` … 식으로 다시 호출.
- `page` 없이도 가능.
- 단, 위 2-1에서 짚은 중복·누락 문제는 그대로 남는다.

장점: 명세 수정 폭이 가장 작다.
단점: "내 주변 실시간 베스트"는 본질적으로 **랭킹 Top-N**이라, 무한히 늘리는 건 의미가 약하다 (베스트 100위 플랜을 누가 본담). B안을 택한다면 사실상 더보기 깊이를 적게(예: 한 번 눌러 5→20) 잡고 끝내는 편이 자연스럽다.

## 4. 권장 명세 변경안

### 4-1. 모든 popular 엔드포인트의 요청 파라미터 통일

| 이름 | 유형 | 필수/선택 | 설명 |
| --- | --- | --- | --- |
| `lat` | number | (위치 기반만) 필수 | MAIN-57·61에서 그대로 |
| `lng` | number | (위치 기반만) 필수 | MAIN-57·61에서 그대로 |
| `limit` | number | 선택 | 페이지당 개수 (기본 5/3/6/5, 최대 50 권장) |
| `page` | number | 선택 | **신규 추가** — 기본 1 (MAIN-57·61에 추가) |

### 4-2. 응답 메타 통일

```jsonc
{
  "status": 200,
  "message": "성공",
  "data": [ /* ...items... */ ],
  "page": 1,
  "limit": 5,
  "hasNext": true,
  "totalCount": 124
}
```

- `hasNext`: 다음 페이지 존재 여부. 무한 스크롤 종료 판단용.
- `totalCount`: 전체 항목 수. 카운트 헤더/페이지네이터 표시용.
- `page`·`limit`은 에코(echo)해 클라이언트가 응답 단위로 상태 검증 가능하도록.

### 4-3. `limit` 상한·검증 정책 명시

- `limit` 최댓값을 명세에 박아둔다 (예: 50). 누락 시 백엔드가 임의로 자를 수 있어 클라이언트가 끝 판정에 혼선이 생긴다.
- `400` 응답 조건에 `limit` 범위 초과를 추가.

### 4-4. (선택) 커서 기반으로의 통일 검토

`PLACE-N4`는 이미 `lastReviewId` 커서다. 실시간 랭킹은 페이지가 깊어질수록 순위 변동으로 오프셋 기반의 약점이 드러나므로, 추후 **커서 기반(`cursor` + `hasNext` + `totalCount`)으로의 통일**도 후보로 둘 만하다. 다만 단기에는 4-1·4-2만으로 충분하다.
