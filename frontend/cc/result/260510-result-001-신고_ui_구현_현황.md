> 작성일: 2026-05-10
> 조사 범위: `frontend/src` 전체 (`신고`, `report`, `Report`, `Flag`, `REPORT_REASONS` 키워드 기준)

---

## 1. 결론 요약

- 프로젝트 내 **신고 관련 UI는 단 한 곳, 플랜 상세 페이지(`CourseViewPage`)에만 구현**되어 있다.
- 별도의 신고 전용 feature 폴더(`features/report/` 등)나 공용 컴포넌트(`shared/ui/ReportDialog` 등)는 **존재하지 않으며**, UI 코드와 상태가 모두 `CourseViewPage.tsx` 내부에 인라인으로 작성되어 있다.
- 백엔드 API 연동도 없다. 신고 제출 시 `toast.success('신고가 접수되었어요.')` 토스트만 띄우고 종료되며, 신고 내용은 어디에도 저장되지 않는다.
- 댓글, 후기, 사용자, 매거진 등 다른 도메인에는 신고 진입점이 **마련되어 있지 않다**.

## 7. 누락/미구현 사항

현재 구현이 **mock UI 수준**에 머물러 있어, 실제 운영을 위해서는 아래 항목들의 추가 구현이 필요해 보인다.

1. **백엔드 연동 부재**
   - 신고 제출 API 엔드포인트(`/api/report` 등)와 axios 호출 로직 없음.
   - `features/course-detail/api/`에도 신고 관련 함수가 없음.
2. **타입 정의 없음**
   - `ReportReason`, `ReportPayload`, `ReportResponse` 등 타입이 미정의.
3. **다른 도메인의 신고 미지원**
   - 댓글(`MOCK_COMMENTS`), 후기, 매거진, 사용자 프로필 등에는 신고 진입점이 없음.
   - 현재 신고 다이얼로그도 "어떤 플랜을 신고하는지" 정보(`courseId`)를 제출 시 포함하지 않음.
4. **재사용성 부재**
   - 신고 UI가 `CourseViewPage` 내부에 인라인으로 박혀 있어, 다른 페이지에서 재사용하려면 공용 컴포넌트(`shared/ui/ReportDialog/` 등)로의 추출이 필요함.
5. **유효성 검증 미흡**
   - 사유 미선택만 검사하고, 상세 내용 길이 제한(`maxLength`)이나 욕설/스팸 필터링 등은 없음.
   - "기타" 사유 선택 시 상세 내용 필수 입력 여부 등의 분기 처리도 없음.
6. **접근성**
   - `<select>`/`<textarea>`에 `aria-label`이 없고, 다이얼로그에 `role="dialog"`/`aria-modal` 속성이 부여되어 있지 않음.

## 1. 신고 대상 및 사유 (기획서 1.1, 1.2 반영)

### 1-1. 대상 (3종 — 기획서 1.1)

| 대상 | 트리거 위치 | 비고 |
| --- | --- | --- |
| **플랜(course)** | 플랜 상세 더보기(⋮) → "신고하기" — 이미 존재 | `courseId`를 페이로드에 실어야 함 (현재 미포함) |

### 핵심 타입

```ts
type ReportTargetType = 'course' | 'user' | 'comment';

interface ReportTarget {
  type: ReportTargetType;
  id: string;
  ownerId: string;       // 자기 신고 차단 + 운영자 검토용
  contextUrl?: string;   // 운영자가 즉시 이동 가능
}

interface ReportPayload {
  target: ReportTarget;
  reasonCode: keyof typeof REASON_LABELS;
  detail?: string;       // ETC 선택 시 필수
}
```

> **사유는 한글 문자열이 아닌 `reasonCode`로 전송**. 통계·다국어·기획 문구 변경에 강함.

## 4. UX 개선

| 현재 | 개선안 | 이유 |
| --- | --- | --- |
| 중앙 모달 (`max-w-xs`) | **BottomSheet** (기존 `ShareBottomSheet` 패턴 재사용) | 한 손 조작, 기존 UI 일관성 |
| `<select>` 드롭다운 | **라디오 리스트** | 모바일 native select는 가독성·접근성 약함 |
| "기타" 선택해도 detail 선택값 | "기타(직접 입력)" 선택 시 **detail 필수** | 기획서 사유 정의와 일치, 운영자 검토 효율 |
| 검증 없음 | `maxLength={500}` + 글자수 카운터 | 도배 방지 |
| 즉시 토스트 종료 | 제출 → 로딩 → 결과 토스트 (성공/이미 신고함/실패 분기) | 네트워크 실패 처리 + 중복 안내 |
| `aria-*` 없음 | `role="dialog"`, `aria-modal`, 포커스 트랩, ESC 닫기 | 접근성 |
| 신고자 비공개 안내 없음 | "신고자 정보는 공개되지 않습니다" 문구 | 신고 부담 완화 |
| 신고 후 화면 그대로 | 신고 직후 **"이 콘텐츠 숨기기"** 옵션 동시 노출 | 자동 블라인드 임계치 도달 전까지 신고자 본인 안전 |

## 5. 백엔드 계약 (제안)

```
POST /api/reports
Body: { target: { type, id }, reasonCode, detail? }
Response:
  201 { reportId, status: 'PENDING', alreadyReported: false, autoBlinded?: boolean }
  200 { alreadyReported: true }     # 중복
  401 unauthorized
  403 self-report-forbidden
  429 too-many-reports              # 중복 신고 횟수 초과 (유저 대상)

GET  /api/reports/me                # 마이페이지: 내가 신고한 내역 + 처리 상태
```

### 주요 동작 보장

- **idempotency-key 헤더**: 더블 클릭/네트워크 재시도 보호
- **rate limit**: 동일 신고자 → 동일 target 24h 1회
- **자동 블라인드 트리거** (기획 2.1): 서로 다른 N명 도달 시 `autoBlinded: true` 반환 → 클라이언트 즉시 숨김 처리
- **상태 enum**: `PENDING → REVIEWING → RESOLVED | REJECTED`

> **TBD 2**: 자동 블라인드 임계치 **N값 미정**. 추천 — 초기 운영은 보수적으로 N=5, 데이터 누적 후 조정.

## 6. 신고 처리 정책 (기획서 2.x 반영)

### 6-1. 플랜 (기획 2.1)

| 단계 | 동작 |
| --- | --- |
| 신고 접수 | `PENDING`. 작성자에게는 알림 X (도배 방지) |
| **자동 블라인드** | 서로 다른 신고자 N명 도달 시 `BLINDED` 상태로 전이 |
| 블라인드 중 | **작성자 권한 제한** — 수정/삭제 불가 (기획서 명시) |
| 운영자 판단 | 삭제(`DELETED`) / 복원(`RESTORED`) |
| 사후 동기화 | **캐시/검색 인덱스 즉시 동기화** (홈/검색 노출 차단) |

> visibility(작성자 제어)와 운영 제어는 **분리된 필드**여야 함. 작성자가 임의로 블라인드 해제 못 하게 함.

### 6-2. 유저 (기획 2.2)

- **중복 신고 제한**: `(reporterId, targetUserId)` 조합으로 횟수/기간 제한.
- **제재 단계** (기획서 명시):

| 누적 경고 횟수 | 조치 |
| --- | --- |
| 1회 | **경고 통보** (1주 내 미조치 시 자동 정지) |
| 2~5회 | 7일씩 누적 정지 (2회=7일, 3회=14일, 4회=21일, 5회=35일) |
| 6회 이상 | **영구 정지** |

## 11. 트레이드오프

### 11-1. 자동 블라인드 임계치 vs 어뷰징

N이 너무 작으면 **타깃성 신고 어뷰징**(특정 사용자에게 다수가 결탁)에 취약. 반대로 너무 크면 임시조치가 늦음. 초기에는 보수적으로 잡고, 운영 데이터로 조정.

### 11-2. 즉시 출시 가능 vs 완벽한 설계

기획서의 제재 시스템 전부를 한 번에 구현하면 출시가 늦어짐. **Phase 1+2까지만 먼저 출시**하고, 자동 블라인드/제재는 신고 데이터가 어느 정도 누적된 후 도입하는 것을 권장.
